# EXBA Framework: Legacy Renderer (V1) Architectural Report

This document records the architecture, module responsibilities, and design patterns of the **Legacy Renderer (`renderer-legacy`)** prior to its cleanup. It serves as historical documentation for the V1 framework prototype.

---

## 🏛️ System Overview

The legacy renderer (`renderer-legacy`) was the V1 prototype of the EXBA browser frontend. It established the baseline custom component model, multi-threaded WASM communication, client-side routing, and signal-based reactivity.

```
   +-------------------------------------------------------------+
   |                  Legacy View Layer (DOM)                    |
   |   (SystemFetchView, StorageView, MapView, AccordionView)    |
   +-------------------------------------------------------------+
                                 |
                                 v
   +-------------------------------------------------------------+
   |                UI Framework & Router (V1)                   |
   |         (BaseComponent, Component, V1 Router)               |
   +-------------------------------------------------------------+
                                 |
                                 v
   +-------------------------------------------------------------+
   |                     Reactivity Layer                        |
   |              (Reactivity.ts, RustState.ts)                  |
   +-------------------------------------------------------------+
            |                                      |
            v (IPC)                                v (Web Worker)
   +-----------------------+              +----------------------+
   |    Electron Main      |              |  Background Worker   |
   |  (SQLite, OS System)  |              |  (WasmWorker.ts)     |
   +-----------------------+              +----------------------+
                                                     |
                                                     v
                                          +----------------------+
                                          |    Rust WASM Core    |
                                          |     (wasm_rust)      |
                                          +----------------------+
```

## 🛠️ Core Modules & Responsibilities

The codebase was divided into several sub-directories under `src/renderer-legacy/core/`:

### 1. WASM Worker Bridge (`core/bridge/`)
To prevent heavy WebAssembly execution from locking up the browser's UI thread, the V1 framework offloaded WASM initialization and method runs to a background Web Worker:
* **`WasmWorker.ts`:** Executed inside a Web Worker thread. It initialized the compiled Rust WASM binary and listened for serialized command buffers, executing them via `process_ir(commandJson)`.
* **`WorkerBridge.ts`:** Managed the lifecycle of the worker thread from the main thread. It implemented a request-response protocol over the `postMessage` API using unique message IDs.
* **`WasmBridge.ts`:** Exposed a clean API facade (`add`, `fibonacci`, `systemFetch`, `greet`) to the UI elements, wrapping the background worker requests in developer-friendly promises.
* **`IRTransformer.ts`:** Handled conversions and validations of JSON structures going to and from the Rust layer.

### 2. Reactivity & State Store (`core/reactivity/`)
* **`Reactivity.ts`:** Contained the V1 signal primitives (`createSignal`, `createEffect`) for tracking reactive dependencies.
* **`RustState.ts`:** Synchronized state changes made in TypeScript with the Rust `StateStore` module.

### 3. Component Spec V1 (`core/ui/`)
* **`Component.ts` & `BaseComponent.ts`:** Defined the initial Custom Element wrapper class. It implemented custom shadow root bindings, V1 template patching, and declarative event delegation (e.g., `'click [data-action="refresh"]': handler`).
* **`ComponentFactory.ts` & `ComponentValidator.ts`:** Handled component registration and enforced basic interface guidelines.
* **`KeyboardManager.ts`:** Exposes global keyboard shortcut listeners for accessibility and hotkey configurations.

### 4. Client-Side Router (`core/routing/`)
* **`Router.ts`:** Managed client-side path transitions and updated the main viewport.
* **`TabManager.ts`:** Handled multiple viewports or tab layouts inside the dashboard workspace.
* **`ServiceRegistry.ts`:** Managed Dependency Injection (DI) for services like WASM, state stores, and database drivers.

### 5. Relational Database Driver (`core/db/`)
* **`FrontendDatabase.ts` & `DatabaseTypes.ts`:** Exposed a wrapper around the Electron IPC channel (`db:execute` and `db:query`) to allow custom views to perform SQL queries on the SQLite backend.

## 📺 Component Views & Demonstrations (`components/views/`)

The legacy renderer had a variety of sandbox views showcasing different capabilities:
* **`SystemFetchView.ts`:** A terminal mockup display showing hardware reports generated by the Rust WASM system fetch compiler.
* **`StorageView.ts`:** Demonstrated SQLite and localStorage operations, verifying schema changes.
* **`MapView.ts` & `ChartView.ts`:** Rendered mapping widgets (Leaflet) and data visualizations (Vega-Lite).
* **`Cube3DView.ts`:** Rendered canvas-based 3D rotations.

## 🔄 Why it was Replaced by V2 (`src/renderer/`)

While the V1 framework was powerful, it was refactored and cleaned up for the following reasons:
1. **API Simplification:** The component model was unified into the clean `ExbaComponent` spec, using static `styles` and `props` definitions to enforce structure warning-free.
2. **Simplified Bundling:** Statically importing WASM via Vite's `new URL('...', import.meta.url)` in `manager.ts` simplified compilation and removed complex worker configuration setups.
3. **Lighter Reactivity:** Bypassing proxy synchronization overhead by relying on reactive signals directly at the view layer.
4. **Improved Test Stability:** The background worker setup in JSDOM tests caused major Vitest timeouts and out-of-memory errors. The V2 codebase resolves this, allowing the test suite to complete in seconds.
